## How this project is operated

This project uses the **project-manager** skill (`{{KIT_PATH}}`). The operating loop is: research → proto-plan → critique (bounded to 2 passes) → decompose → build → log → review.

The bounded-iteration rule is enforced: the agent and the human stop critiquing after pass 2 even if it still feels imperfect. AI will happily generate plausible new blindspots forever — that's a planning trap. A polished plan is not the goal; a working project is.

## Commands you can suggest the user run

- `pm status` — current state, blockers, recent activity.
- `pm what-now` — top 3 ranked next actions with justifications.
- `pm log "<entry>"` — append a session log entry.
- `pm log --decision "<entry>"` — append to the decisions ledger.
- `pm brief` — refresh the investor-readable project brief.
- `pm critique <plan-file>` — run the bounded 2-pass critique on a plan or subplan.
- `pm decompose <plan-file>` — split a plan into subplans.

## Conventions

- **Session logs are cheap.** Log liberally. Future-you and the next collaborator both depend on it.
- **The plan is provisional.** When the build teaches you the plan was wrong, edit the plan. Don't apologize.
- **Critique passes are bounded.** Two passes per section. Pass 3 is refused by the tooling.
- **One thing at a time.** When in doubt, ship the smallest version that works.
