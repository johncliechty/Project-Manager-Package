# Milestone 3 — Scaffold — Acceptance Rubric

**Intent (one sentence the student also reads):**
You have written a single `index.html` file that loads in your browser and shows Anchor's empty shell — the four regions from your M2 picture visible on the page, styled sensibly, with the five data nouns declared as variables in a `<script>` block that does nothing else.

## The boxes

Read each box. For each, check the project state. Mark it ✓ or ✗. **All eight must be ✓ to advance to M4.**

- [ ] `index.html` exists at the project root (not inside `.project-manager/`, not inside a subfolder — at the root of the project, next to the `.git/` folder and the `.project-manager/` folder).
- [ ] **The page renders in a browser.** Double-clicking `index.html` opens a browser tab showing the page without errors. The browser's developer console (open with `Ctrl+Shift+J` on Windows, `Cmd+Option+J` on Mac) shows zero JavaScript errors. If the page looks broken or the console is red, this box is not green — diagnose the typo with the student, climbing the hint ladder.
- [ ] **The page contains the four named regions from the M2 picture, as real structural HTML elements.** A header (e.g., `<header>` with an `<h1>`). A project sidebar (e.g., `<aside>` or `<div class="sidebar">` containing a `<ul>` with at least one `<li>` for "Personal"). A task list region (e.g., `<div id="tasks">` or `<ul id="tasks">`, currently empty). An input for adding tasks (`<input>`, optionally inside a form-style container). Each region must be an actual HTML element, not a paragraph of description (`<p>Sidebar with projects: Personal</p>` is **not green** — that's prose pretending to be structure).
- [ ] **The `<script>` block declares the five data nouns from M2 as variables, and nothing else.** The five typical for Anchor: `tasks`, `projects`, `selectedProject`, `currentDraft`, and one more from the student's M2 picture (often a counter like `nextId` or a flag like `showingCompleted`). Names are the student's choice; what matters is each M2 data noun has a corresponding `let` (or `const`) declaration with a sensible empty default (`[]`, `""`, `null`, etc.). **There is no other JavaScript in the file.** No functions. No event listeners. No `console.log`. No `addEventListener`. If the student wrote any working logic, this box is not green — strike it now, M4 brings it back legitimately.
- [ ] **The styling is sufficient that the layout matches the M2 picture.** The sidebar is visibly on one side; the task list is visibly on the other; the header sits on top; the input sits on the bottom. The font is readable (a system-font stack like `system-ui, -apple-system, sans-serif`). There is enough whitespace that the page doesn't feel cramped. An accent color exists (whatever the student picked). **It does not need to be pixel-perfect** — it needs to be *recognizable* as the M2 mockup. Polish lives in M4.6.
- [ ] **The student looked at multiple concrete options, made a deliberate choice, directed the agent through at least one round of refinement, and *owns the file*.** "Owns the file" means two things. First, the student can explain *why they picked what they picked* — *"I picked Option A because the minimal feel matched the brief, but I asked for Option C's month organization because I wanted the structure."* Second, the student can answer, in plain English, what each region of the HTML does (*"that's the sidebar that holds my projects"*) and what each variable in the script block holds (*"`tasks` is the list of all tasks; `selectedProject` is which project I'm currently looking at"*). The coach tests both during the checkbox sweep — asking *"why this option and not the others?"* once and 2–3 specific *"what does this part do?"* questions. If the student can't answer, walk through that part of the file with them — pointing at lines, explaining what each one does — until they can. **The student does not need to have typed any of the code.** Ownership is by *choice* (looking at multiple options, articulating the trade-offs) and *understanding* (knowing what's in the file you settled on), not by keystroke. Choosing among concrete options forces the student to articulate their criteria, which is the metacognitive scaffold that turns a passive viewer into an active engineer.
- [ ] **The student has received a structured review of `index.html` from the coach, and every suggested edit has been either applied or replied to with a one-sentence explanation of why not.** This is the M3-closing handshake (Practice 7). The review opens with what is already working (by line — *"the file renders without errors and the shape is reachable"*), proposes 2–4 specific edits with one-sentence justifications grounded in the M3 rules above (structural elements not descriptive paragraphs, empty script, five data nouns declared, layout matches the M2 picture), and closes with an invitation to do another round if the student wants to push the shell further. The gate is **engagement, not agreement.** See `references/STYLE.md` § *How to review a student's brief* — the four-rule lens generalizes by substituting the M3 rules for the M1 Lecture rules.
- [ ] **Third git commit has happened** with the message `M3: empty shell loads`. If the shell changed substantially after the review, an optional follow-up commit `M3.1: shell revisions from review` reflects the post-review state. The student typed the commit themselves (or said yes to the auto-commit prompt on the default-on path).

## How you tell the student where they are

If 6–7 boxes are green and 1–2 are red, say specifically which ones and offer a one-line next action. Example:

> *"You're close on M3. The page renders cleanly and the four regions are there. Two things still red: (1) the `<script>` block has an `addEventListener` line that's doing work — strike it, M3 is the empty stage; (2) the sidebar shows "Personal" but the surrounding `<aside>` is missing — wrap the list in `<aside class="sidebar">`. We're a small edit away."*

If fewer than 6 boxes are green, the milestone needs more work. Pick the box that unlocks the next two and work on that. For M3 the most common red is **box 4 (working JavaScript snuck in)** — the temptation to make something work is strong, and the discipline of resisting it is the lesson.

**Ordering note for the review handshake.** Boxes 1–6 (the file content checks plus ownership) must be green *before* the review happens — reviewing a file that doesn't load isn't a review, it's a debugging session. The natural flow matches the Practice sequence in Lesson 3: student describes what they want based on `picture.md` (Practice 1) → agent produces first draft, opens it in browser (Practice 2: student looks) → student gives 2–3 specific feedback items (Practice 3) → agent updates; student iterates until satisfied (Practice 4) → coach walks the rubric and tests ownership (Practice 5: checkbox sweep) → optional friend interaction (Practice 6) → coach gives the M3-closing structured review (Practice 7) → student applies or explains each suggested refinement → review-received box green → third git commit `M3: empty shell loads` → optional follow-up commit if review-edits changed the file → all eight green → advance to M4.

If all eight are green, mark M3 complete in `.project-manager/state.json`, propose the commit (and the optional follow-up if needed), congratulate the student in one sentence, and open M4.

## Why this rubric

M3 is the milestone where the student's *role* changes. In Lessons 1 and 2 the student was the **writer** — the brief and the picture are prose, and prose is something only the student can produce because only they know what their app should be. From M3 onward the student is the **director** — the agent writes the code based on the student's description, the student looks at the result, gives feedback, and iterates. Same load-bearing engineering frame ("the student is the engineer; the agent does the work"), inverted seniority for code-producing milestones.

The "ownership through understanding" rule for box 6 is the firmest of any rubric so far. The agent could produce a clean, working `index.html` in two seconds and call it done — pedagogically that would be a disaster, because the student would have no model of why the file works or how to fix it when it breaks. The discipline is the **iteration loop**: describe → look → feedback → look again → iterate. Each round forces the student to *see* what's in the file. By the end of two or three rounds, the student has looked at every region, every variable, and every styling choice — and decided whether each one matches their intent. *That* is ownership. Typing was never the point; understanding was.

The "no working JavaScript yet" rule is the rubric's other load-bearing constraint. M3's whole point is *separating structure from behavior.* A student who lets the agent wire up `addTask` in M3 has skipped M3 and started M4 with one foot still on the M3 floor — they don't have a known-good shell to add features onto. The agent (which is also playing the coach role) holds this line firmly: *"I noticed I drifted into M4 territory — let me strike that function. It comes back in M4.1, on a clean foundation."*

The "match the M2 picture" rule is what makes the M2 work pay off. If the M3 shell looks nothing like the M2 mockup, M2 was theater. The rubric checks the structural match (the four regions in the right places) but not the *aesthetic* match (the exact color, the exact spacing). Aesthetic precision is M4.6's job; M3 just needs to be *recognizable* as the picture the student described.

The director's loop runs five to six rounds in a typical M3 session. The agent should be watching for **convergence** — if the student is still asking for big structural shifts after round three, something's wrong with the initial description and the agent should back up and reread `picture.md` with the student. Convergence is also how the agent knows when the student is ready for the M-closing review: rounds are getting smaller, the student is satisfied, the rubric is going green naturally.

The review handshake at the end starts to feel familiar by now. By M3 the student has received structured review on a brief, on a picture, and now on a code file they directed into existence. They're learning the shape: highlights first, specific refinements grounded in rules, an invitation to iterate. By M5 they'll start anticipating what the coach is about to flag, and that anticipation is the engineering-judgment skill the whole curriculum is training. This is what real software work looks like in 2026, taught on a project where the stakes are exactly the right size.
